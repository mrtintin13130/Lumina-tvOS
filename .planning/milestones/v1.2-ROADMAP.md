# Roadmap: Lumina Native tvOS Client v1.2

## v1.2 Stability, Usability, And Beta Hardening

## Overview

This roadmap turns the current native tvOS client from a promising playback-oriented app into a stable, usable, remote-first beta candidate. The milestone deliberately avoids broad feature expansion. It focuses on the highest-risk issues found in senior review: central state concentration, unfinished user-visible actions, uneven tvOS focus behavior, playback lifecycle proof, useful safe diagnostics, and repeatable verification.

- Decimal phases (11.1, 11.2): Urgent insertions if needed.

## Phase Details

### Phase 11: Architecture And State Hardening

**Goal**: Developers can extend and debug the app without routing every workflow through one central `AppModel`.
**Mode:** mvp
**Depends on**: v1.1 proof-path implementation and current unit tests
**Requirements**: [ARCH-01, ARCH-02, ARCH-03, ARCH-04]
**Success Criteria** (what must be TRUE):

1. Session, catalog, playback, and diagnostics responsibilities are separated into small models/services or coordinators with clear ownership.
2. Existing setup, sign-in, Home, detail, search, settings, and playback behavior remains intact.
3. Stale async results cannot mutate the wrong screen after sign-out, navigation, search changes, detail changes, editorial changes, or playback cancellation.
4. Token and diagnostics security rules remain covered by tests.

**Plans**: 4 plans

Plans:

- [x] 11-01: Extract session/auth and server state ownership from `AppModel`.
- [x] 11-02: Extract catalog/search/detail/editorial state ownership with stale-load guards.
- [x] 11-03: Extract playback lifecycle state and progress/session reporting ownership.
- [x] 11-04: Add focused regression tests for state transitions, stale loads, sign-out, and security boundaries.

### Phase 12: Remote-First tvOS UX Hardening

**Goal**: The app feels intentional from the couch, with visible focus, predictable remote navigation, and no polished dead-end controls.
**Mode:** mvp
**Depends on**: Phase 11
**Requirements**: [UX-01, UX-02, UX-03, UX-04, UX-05]
**Success Criteria** (what must be TRUE):

1. Every primary screen and overlay has a sensible default focus target.
2. Returning from details, editorial overlays, search results, settings, and playback restores focus where users expect it.
3. Placeholder actions are hidden, disabled, or clearly marked unavailable before selection.
4. Setup and search are less painful with remote text entry and provide clear recovery states.
5. Async artwork, focus scale, shelves, and overlays do not cause layout jumps or unreadable text at TV distance.

**Plans**: 4 plans

Plans:

- [x] 12-01: Audit and fix default focus plus focus restoration across setup, Home, grids, search, detail, editorial, and settings.
- [x] 12-02: Hide or safely disable unfinished episode, trailer, genre, and person actions.
- [x] 12-03: Improve setup/search text-entry ergonomics, normalization feedback, recent/saved server affordances, and empty/error states.
- [x] 12-04: Run visual/focus QA on major screens and stabilize text, artwork, safe-area, and async loading behavior.

### Phase 13: Playback Lifecycle Reliability Proof

**Goal**: Physical Apple TV playback is trustworthy across normal use and known failure cases.
**Mode:** mvp
**Depends on**: Phase 11
**Requirements**: [PLAY-21, PLAY-22, PLAY-23, PLAY-24, PLAY-25]
**Success Criteria** (what must be TRUE):

1. A physical Apple TV can start a movie, pause/scrub where supported, exit, stop, relaunch, and resume against a live server.
2. Backend playback session and progress records confirm the expected cadence and state transitions.
3. Expired token, HLS failure, missing media, server restart, and unreachable server states produce safe user-facing recovery behavior.
4. Audio/subtitle availability is compared across backend tracks, HLS manifests, and AVKit media selection where representative media exists.
5. Playback cleanup cannot leave stale app state or stale backend sessions.

**Plans**: 5 plans

Plans:

- [x] 13-01: Harden playback state transitions for pause, exit, stop, failure, relaunch, and resume.
- [x] 13-02: Verify progress/session cadence and backend evidence for normal playback.
- [x] 13-03: Verify expired token, HLS failure, missing media, server restart, and unreachable server recovery.
- [x] 13-04: Verify audio/subtitle availability and safe media-selection diagnostics on representative physical media.
- [x] 13-05: Capture safe physical Apple TV proof evidence and record follow-up gaps.

### Phase 14: Safe Diagnostics And Support Surface

**Goal**: Settings provides useful support context without becoming analytics or leaking sensitive material.
**Mode:** mvp
**Depends on**: Phase 11 and Phase 13 findings
**Requirements**: [DIAG-01, DIAG-02, DIAG-03, DIAG-04]
**Success Criteria** (what must be TRUE):

1. Settings shows app/build, server validation, server capability/version summary, signed-in user display name, and last safe error.
2. Diagnostics events can correlate with backend support data through safe IDs and categories.
3. Redaction covers token-like values, signed URLs, credentials, local paths, SQL details, stack traces, and raw subprocess output.
4. Diagnostics remain local/support-oriented and do not add third-party telemetry or monetization analytics.

**Plans**: 3 plans

Plans:

- [x] 14-01: Design and implement the safe Settings support information surface.
- [x] 14-02: Expand structured diagnostics fields for support correlation without secret leakage.
- [x] 14-03: Add redaction/support-surface tests and update diagnostics documentation.

### Phase 15: Verification And Small-Beta Readiness

**Goal**: The team has repeatable confidence gates and a clear small-beta path.
**Mode:** mvp
**Depends on**: Phases 11-14
**Requirements**: [VER-11, VER-12, VER-13, VER-14, VER-15]
**Success Criteria** (what must be TRUE):

1. Generic tvOS build and build-for-testing pass using DerivedData outside fragile user-local Xcode state.
2. Unit tests cover the new architecture split, playback cleanup, diagnostics, and security redaction.
3. XCUITests cover setup/sign-in, Home, search, detail, playback entry, Settings, and sign-out smoke paths.
4. Physical Apple TV QA matrix is filled with safe evidence and known residual risks.
5. TestFlight beta readiness notes identify reviewer path, demo server/media, privacy decisions, assets, deferrals, and release blockers.

**Plans**: 4 plans

Plans:

- [x] 15-01: Standardize repeatable build, build-for-testing, and local verification commands.
- [x] 15-02: Replace generated UI tests with focused tvOS smoke coverage.
- [x] 15-03: Complete physical Apple TV QA matrix and evidence review.
- [x] 15-04: Update TestFlight beta readiness docs, reviewer path, privacy notes, and known deferrals.
