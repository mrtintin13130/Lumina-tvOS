# Concerns

## Summary

The codebase is in an early scaffold state. The main risk is not existing code complexity; it is the gap between the generated SwiftUI placeholder and the ambitious tvOS client described in `TVOS_CLIENT_PRD.md`. Early implementation should establish secure, testable boundaries for networking, auth, playback, diagnostics, and tvOS focus before feature breadth expands.

## Product-To-Code Gap

- `TVOS_CLIENT_PRD.md` describes server setup, JWT auth, catalog browsing, search, details, HLS playback, progress reporting, watchlist/favorites, settings, diagnostics, and physical Apple TV validation.
- The current app code in `lumina/ContentView.swift` is still a static "Hello, world!" view.
- There are no models, services, API clients, storage layers, or playback components yet.

Impact: planning and implementation should avoid assuming any supporting infrastructure already exists.

## Auth And Secret Handling Risk

The PRD requires Keychain token storage and strict redaction, but no auth layer exists yet.

Risks to watch:

- JWTs or stream tokens leaking into logs.
- Authorization headers appearing in diagnostics.
- Stream tokens being exposed in user-visible errors.
- Password handling being implemented directly in SwiftUI views.
- Token persistence being implemented with an insecure store instead of Keychain.

Relevant requirements live in `TVOS_CLIENT_PRD.md` under Authentication and Security And Privacy Requirements.

## Playback Risk

Playback is the highest-risk workflow:

- AVKit/HLS behavior must be proven on physical Apple TV.
- Simulator success is not enough.
- HLS manifests, segment errors, token expiry, unsupported tracks, and missing media need user-safe error states.
- Progress reporting and resume behavior must align with backend state.

No playback code exists yet, so the first vertical slice should stay narrow and test the full server-auth-stream-progress path before broad UI work.

## tvOS Focus And Remote-Control Risk

The current `ContentView` does not exercise focus movement, remote input, long rows, forms, or player transitions.

Future risk areas:

- Focus landing on invisible or disabled content.
- Search/browse pagination causing focus jumps.
- Remote-control text entry making username/password sign-in painful.
- Phone-style layouts failing on a 10-foot interface.

`TVOS_CLIENT_PRD.md` calls out focus and navigation requirements that should be treated as implementation constraints.

## Testing Risk

- `luminaTests/luminaTests.swift` contains generated placeholder tests only.
- `luminaUITests/luminaUITests.swift` only launches the app.
- `luminaUITests/luminaUITestsLaunchTests.swift` captures a launch screenshot.
- There are no assertions for app behavior.
- There is no CI configuration.

Generated tests should be replaced or supplemented quickly once real features land.

## Xcode Environment Concern

During mapping, `xcodebuild -list -project lumina.xcodeproj` completed and listed the expected targets and scheme, but emitted CoreSimulatorService, file coordination, and log permission warnings in the sandboxed environment.

Impact:

- Project metadata is readable.
- Command-line simulator or UI-test runs may fail until run in a normal local Xcode environment or until simulator services are healthy.

## Repository Hygiene Concern

- `TVOS_CLIENT_PRD.md` is untracked according to `git status --short`.
- `.planning/codebase/` is newly generated by this workflow.
- `lumina.xcodeproj/project.xcworkspace/xcuserdata/martin.xcuserdatad/UserInterfaceState.xcuserstate` is user-local Xcode UI state and is usually not desirable in shared version control.
- `scripts/` exists but is empty.

Before broader collaboration, consider adding or reviewing `.gitignore` rules for Xcode user state and generated local artifacts.

## Architecture Boundary Concern

Because the scaffold has no service or feature boundaries, early code may drift into a large `ContentView` or view-heavy implementation.

Watch for:

- Network calls directly in SwiftUI views.
- Token storage accessed directly from views.
- Playback session logic mixed into player presentation.
- Error mapping duplicated across screens.
- DTOs and domain state becoming indistinguishable.

Early separation of API clients, auth/session state, playback coordination, and diagnostics will reduce rework.

## Backend Contract Concern

The PRD references many candidate backend routes, but the current tvOS repo does not contain the backend implementation.

Risks:

- Client assumptions may diverge from actual Lumina API behavior.
- Capability negotiation may be skipped.
- Unsupported filters or fields may leak into UI.
- Backend error envelopes may not match client decoding.

Mitigation: build typed DTOs from real backend responses and validate `/api/v1/system/capabilities` before enabling dependent UI.

## Asset And Branding Concern

The tvOS asset catalog has generated app icon and Top Shelf image structure, but likely placeholder content.

Risks:

- App Store readiness will require real layered tvOS icons and Top Shelf assets.
- Top Shelf product behavior is deferred, but asset readiness is still part of release preparation.

## Performance And Memory Concern

No performance-sensitive code exists yet, but future Home/catalog screens and playback surfaces will need care:

- Large libraries can make row rendering and image caching expensive.
- Missing or partial artwork must not create layout instability.
- Playback plus diagnostics should avoid unnecessary memory pressure.
- Long playback sessions need monitoring beyond launch performance tests.

## Security Scan Note

This mapping did not find implemented secrets in source code. The generated codebase documents should still be scanned before commit because they summarize planned auth, stream-token, and API behavior.
